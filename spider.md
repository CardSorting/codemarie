§ 01
Core Concept & Problem Statement
Every software application has an architecture — a shape defined by its routes, resources, domains, and navigation paths. Today, that shape is an emergent artifact: it exists only as a side-effect of code, and it drifts toward chaos through thousands of small unchecked decisions.

Spider inverts this. Structure is defined first, as a formal spec. Code must conform to it. Violations surface immediately — in the editor, in CI, and on every deploy. Entropy is measured and tracked. Structural history is versioned. Architecture becomes enforceable, inspectable, and auditable for the first time.

The One-Sentence Definition
Spider is a linter for your application's architecture — not its code, not its types, but the shape of the system itself.

What Spider Is Not
Spider is not a router, not a framework, not a code generator (yet), and not a documentation tool. It does not care what language or framework you use. It operates on a structural model of your application that is framework-agnostic. All it needs is the spec and a way to observe the current state of your routes and resources.

Hard Design Constraint
A developer who has never heard of Spider must be able to read spider.spec.json and understand it in under 60 seconds. If the spec is not immediately legible to a new reader, the design has failed.

§ 02
Stack Position & Scope Boundaries
Spider occupies a new layer between the framework layer and the type layer. No existing tool covers this scope:

Tool	Scope	Hierarchy-aware	Semantic	Evolution
ESLint	Syntax	No	No	No
TypeScript	Types	Partial	No	No
OpenAPI	API shape	No	No	No
Next.js	File conventions	Partial	No	No
🕷️ Spider	Architecture	Yes	Yes	Yes
Spider's input is spider.spec.json plus a structure source — the current observable state of the application's routes and resources. The structure source is pluggable: it can be filesystem crawl, a manifest generated by the framework, or a programmatic API call. Spider does not care how code is organized internally.

§ 03
The Spec DSL MVP
The spec is a single spider.spec.json file at the project root. It is the authoritative source of structural truth. Everything else in Spider derives from it.

Top-Level Schema
Field	Type	Req?	Description
version	semver string	REQ	Spec format version. Used to detect breaking schema changes.
domain	string	REQ	Canonical domain identifier for this application. Used in evolution snapshots.
rules	RuleSet	REQ	Global enforcement rules applied to all resources.
resources	ResourceMap	REQ	Named map of top-level structural resources.
domains	DomainMap	OPT	Named separation boundaries (e.g. ui, api, admin). Enables cross-domain leak detection.
invariants	string[]	OPT	Natural-language invariant assertions. Compiled to constraint checks at boot.
meta	object	OPT	Free-form metadata (author, team, links). Not processed by Spider.
RuleSet Schema
Field	Type	Default	Description
maxDepth	number	4	Maximum nesting depth of any resource path. Depth is counted from root. Violation: ERROR.
naming	NamingScheme	"kebab"	Naming convention for path segments. One of: kebab, camel, snake, plural-kebab.
noOrphans	boolean	true	Disallow routes with no inbound navigation edges in the structural graph.
noCircularDeps	boolean	true	Disallow cycles in the structural graph.
maxSiblings	number	10	Maximum children per resource node. Exceeding suggests missing sub-grouping.
requireParentForId	boolean	true	Any :id resource must have a list parent (e.g. /posts/:id requires /posts).
separateDomains	string[]	[]	Domain names that must not have cross-references. Violation: ERROR.
entropyThreshold	number	0.75	Entropy score above which Spider emits a WARN on check runs.
entropyHardLimit	number	0.90	Entropy score above which Spider exits non-zero.
Resource Node Schema
Field	Type	Req?	Description
path	string	REQ	URL path. Supports :param syntax. Must begin with /.
intent	ResourceIntent	OPT	Semantic label: collection, detail, action, modal, admin, auth. Used in invariant checks.
domain	string	OPT	Domain assignment. Must match a key in top-level domains. Inherits from parent if unset.
access	string	OPT	Free-form access label (e.g. owner-only, public). Not enforced — used in visualization and docs.
children	ResourceMap	OPT	Nested child resources. Recursive. Depth counter increments per nesting level.
navigatesTo	string[]	OPT	Explicit navigation edges to other resource paths. Used to build the structural graph.
deprecated	boolean	OPT	Marks route for removal. Spider emits WARN if deprecated routes persist past deprecatedAfter.

§ 04
Constraint Engine MVP
The Constraint Engine is the runner that validates the current application state against the spec. It operates on two inputs: the parsed spec and a StructureSource — the live or crawled representation of actual routes.

Violation Type System
Severity	Behavior	CI behavior
ERROR	Hard invariant violated. Route missing a required parent, naming convention broken, domain leak, circular dep.	Exit code 1. Blocks merge/deploy.
WARN	Soft structural issue. Orphaned route, deprecated route still present, entropy above threshold, sibling count approaching limit.	Exit code 0. Annotates PR. Does not block.
INFO	Advisory. New resource added, depth increased, structural diff available.	Exit code 0. Logged only.
Violation Object Shape
Violation

id
string
Unique violation code. e.g. SPI-001, SPI-042. Stable across versions — never renumber.
severity
"ERROR"|"WARN"|"INFO"
Severity level.
rule
string
The rule key that triggered this violation. e.g. requireParentForId.
path

The resource path this violation applies to. Null for global violations.

Human-readable description. Should suggest a fix, not just describe the problem.

Arbitrary data relevant to the violation (e.g. current depth, allowed depth).
Rule Registry Pattern
Rules are registered individually, not hardcoded into a monolithic checker. This enables custom rules and plugin architecture in later versions.

// Rule interface — every built-in and custom rule implements this
interface SpiderRule {
  id: string             // e.g. "SPI-007"
  name: string           // e.g. "requireParentForId"
  defaultSeverity: Severity
  check(ctx: RuleContext): Violation[]
}

// RuleContext passed to every rule check()
interface RuleContext {
  spec: SpiderSpec           // parsed spec
  source: StructureSource    // live/crawled structure
  graph: StructuralGraph     // built before rules run
  snapshot: Snapshot | null  // previous snapshot for diff rules
}
Built-in Rule Codes (MVP)
Code	Rule name	Default	Description
SPI-001	maxDepth	ERROR	Path exceeds maxDepth rule value.
SPI-002	naming	ERROR	Path segment violates naming scheme.
SPI-003	requireParentForId	ERROR	:id resource has no collection parent.
SPI-004	separateDomains	ERROR	Cross-domain navigation edge between separated domains.
SPI-005	noCircularDeps	ERROR	Cycle detected in structural graph.
SPI-006	noOrphans	WARN	Route has no inbound edges and is not a root.
SPI-007	maxSiblings	WARN	Resource has more siblings than maxSiblings.
SPI-008	deprecatedExpired	WARN	Deprecated route has passed its deprecatedAfter date.
SPI-009	entropyThreshold	WARN	Entropy score exceeds soft threshold.
SPI-010	entropyHardLimit	ERROR	Entropy score exceeds hard limit.
SPI-011	undeclaredRoute	WARN	Route exists in source but not in spec.
SPI-012	missingRoute	ERROR	Route declared in spec but not found in source.
§ 05
Structural Graph V2
The structural graph is a directed multi-edge graph built from the spec. It is constructed once per run before any rules execute, and passed into every rule's context. Graph-dependent rules (orphan detection, cycle detection, domain leak) cannot run without it.

Implementation note
Use an adjacency-list representation. Do not use a matrix. Resource count in real applications is typically 20–500 nodes — adjacency list is the correct choice for this scale. Recommended library: none. Implement the traversal directly — the graph operations required (DFS, BFS, cycle detection) are simple enough that a dependency is unjustified.

Required Graph Algorithms
Algorithm: Cycle Detection (DFS-based)
Initialize visited set and recursion stack.
For each unvisited node, run DFS.
On entry, add node to recursion stack.
For each neighbor: if in recursion stack → cycle found. Record path. If unvisited → recurse.
On exit, remove node from recursion stack.
Return all detected cycles as arrays of node ids.
Rule SPI-005 generates one ERROR per detected cycle, with the full cycle path in context.
Algorithm: Reachability / Orphan Detection
Identify root nodes: all nodes with
depth === 0
OR nodes with no inbound edges of type
navigation
or
parent-child
.
Run BFS/DFS from each root, traversing all outbound edges.
Any node not reached is unreachable.
Unreachable nodes that are not explicitly
deprecated
→ Rule SPI-006 WARN.
Note:
action
edges do not count for reachability (they represent API calls, not navigation).
Algorithm: Depth Measurement
Depth is computed at graph-build time by counting
/
-separated segments in the normalized path, minus 1 for root.
/
→ depth 0.
/projects
→ depth 1.
/projects/:id/settings
→ depth 3.
Store as
GraphNode.depth
. Rule SPI-001 reads this directly — no re-traversal needed.
Also compute
avgDepth
and
maxObservedDepth
as graph-level metrics used in entropy scoring.
§ 06
Entropy Engine V2
The entropy score is a single normalized value in [0.0, 1.0] that represents the structural health of the application. 0.0 is perfectly clean. 1.0 is complete chaos. The score is computed from a weighted sum of observable structural properties.

Design principle
The entropy score formula must be deterministic, documented, and stable across Spider versions. Teams will use it as a PR gate. If the formula changes, the version number changes and a migration note is published. Never silently change the scoring.

Entropy Score Formula v1
EntropyScore — Formula v1.0
E = ( w₁ × D + w₂ × N + w₃ × O + w₄ × P + w₅ × C )

where:
D = depthScore w₁ = 0.30
N = namingScore w₂ = 0.20
O = orphanScore w₃ = 0.20
P = patternScore w₄ = 0.15
C = couplingScore w₅ = 0.15
D
avgDepth / maxDepth (clipped to 1.0)
N
namingViolations / totalNodes (clipped to 1.0)
O
orphanNodes / totalNodes (clipped to 1.0)
P
duplicatePatternPairs / totalNodes (clipped to 1.0)
C
crossDomainEdges / totalEdges (clipped to 1.0)
Duplicate Pattern Detection
Two resources are considered a duplicate pattern if their normalized path templates are identical siblings under different parents — e.g. /projects/:id/settings and /accounts/:id/settings define the same shape but have no shared invariant in the spec. This suggests missing abstraction.

Algorithm: Duplicate Pattern Detection
For each node, extract its path template: replace all
:param
values with
:p
, remove the parent prefix.
Group nodes by template string.
For each group with size > 1: check if they share a common parent in the spec. If not, record as a duplicate pattern pair.
patternScore = duplicatePairCount / totalNodes (clipped to 1.0).

§ 07
Evolution Engine V3
The Evolution Engine maintains a versioned history of structural snapshots. On each spider check run with --snapshot, Spider writes a snapshot to .spider/snapshots/. Diffs are computed between adjacent snapshots to produce a structural changelog.

Snapshot Format
{
  "spiderVersion": "1.0.0",
  "timestamp":    "2026-03-20T14:22:00Z",
  "commitHash":   "a3f7c91",         // optional — from git env
  "entropyScore": 0.61,
  "nodeCount":    24,
  "edgeCount":    31,
  "avgDepth":     2.1,
  "nodes": [
    { "id": "/projects/:param", "depth": 2, "domain": "ui", "intent": "detail" },
    ... (all nodes)
  ],
  "edges": [
    { "from": "/projects", "to": "/projects/:param", "type": "parent-child" },
    ... (all edges)
  ]
}
Diff Algorithm
Algorithm: Structural Diff (A → B)
Build node sets
A.nodes
and
B.nodes
by id.
Added nodes: ids in B but not A. Tag as
+
.
Removed nodes: ids in A but not B. Tag as
-
.
Moved nodes: nodes where id changed but
label
is identical and edit-distance of paths is below threshold. Tag as
~
.
Changed nodes: same id but different depth, domain, or intent. Tag as
!
.
Repeat steps 2–5 for edges.
Compute entropy delta:
B.entropyScore - A.entropyScore
. Positive = degradation.
Output as structured StructuralDiff object, rendered as human-readable diff in CLI output.
Example structural diff output
+ /projects/:id/members          // added — correctly nested
~ /projects/:id/prefs → /projects/:id/settings  // moved
- /users                          // removed
! entropy delta: +0.04           // slight degradation
Storage
Snapshots are written to .spider/snapshots/{timestamp}-{shortHash}.json. The .spider/ directory should be committed to version control. The most recent N snapshots (default: 50) are retained; older ones are pruned automatically. A .spider/history.json index file maintains the ordered list of snapshots with metadata for fast lookup.

§ 08
CLI Surface
Spider ships as a CLI tool installable via npm. The binary name is spider. All commands share a common config resolution: --config flag > spider.spec.json in cwd > walk up to project root.

spider init
Scaffold a spider.spec.json interactively. Detects framework (Next.js, Remix, etc.) and offers to auto-populate resources from crawling the filesystem. Writes the spec and a .spider/ directory.
spider check
Run the full constraint engine against current structure. Prints violations. Exits 0 if no ERRORs, exits 1 if any ERROR. Accepts --json for machine-readable output, --snapshot to write a snapshot, --fail-on-warn for stricter CI.
spider graph
Output the structural graph as JSON or as a Mermaid diagram string. Accepts --format json|mermaid|dot. Used by the IDE extension and by CI to render PR annotations.
spider entropy
Compute and print the current entropy score with component breakdown. Accepts --json. No snapshot is written unless --snapshot is passed.
spider diff [ref]
Compute structural diff between current state and a previous snapshot. ref is a snapshot timestamp or commit hash. If omitted, diffs against the most recent snapshot.
spider history
Print the snapshot history with entropy scores over time. Accepts --last N to limit output. Useful for spotting the commit where entropy spiked.
spider watch
Watch mode. Re-runs spider check on file changes to the spec or to route files. Used during active development. Outputs to stdout with clear/reprint on each run.
Exit Codes
Code	Meaning
0	All checks passed. No ERRORs. WARNs may be present.
1	One or more ERROR-severity violations found.
2	Spider configuration error (spec not found, spec invalid JSON, schema mismatch).
3	Structure source error (crawler failed, no routes found).
§ 09
Programmatic API
The CLI is a thin wrapper over a programmatic Node.js API. All functionality is exposed for use in build tooling, test runners, and the IDE extension.

import { Spider } from '@spider-eng/core'

// Initialize from spec file
const spider = await Spider.fromFile('./spider.spec.json')

// Run constraint check against a custom structure source
const result = await spider.check({
  source: myCustomStructureSource,  // implements StructureSource interface
  snapshot: true
})

result.violations   // Violation[]
result.graph        // StructuralGraph
result.entropy      // EntropyReport
result.exitCode     // 0 | 1 | 2 | 3

// Structural graph as Mermaid string
const diagram = spider.graph.toMermaid()

// Diff against previous snapshot
const diff = await spider.diff({ ref: 'latest' })

IDE Extension V3
The VS Code extension (VSIX) is Spider's visual control surface. It communicates with the Spider CLI and programmatic API over a local JSON-RPC socket. The extension does not re-implement any Spider logic — it is a thin UI layer.

Extension Features

Graph panel	V2	Webview panel rendering the structural graph as an interactive D3 force graph. Click node → jump to route file. Filter by domain.
Entropy status bar	V2	Live entropy score in the status bar. Color-coded: green / yellow / red. Click → open entropy breakdown panel.

Build the extension only after the CLI and programmatic API are stable. The extension must have no business logic — it calls the Spider API and renders results. This keeps the extension thin and testable. The JSON schema for spider.spec.json should be published to SchemaStore as the first distribution step.

§ 12
Risks & Sharp Edges
Risk
Level
Mitigation
Spec becomes stale — developers stop updating it and it drifts from reality
HIGH
Rule SPI-011 (undeclared routes) and SPI-012 (missing routes) are the defense. Make keeping the spec current a required part of the PR process — not a suggestion.
Over-engineering — Spider itself becomes too complex to adopt
HIGH
Phase gates enforce this. The 60-second spec readability test is non-negotiable. Each phase must be in daily use before the next begins.
Framework diversity — adapters for non-Next.js frameworks are hard to maintain
MED
The StructureSource interface is designed for community-contributed adapters. Start with Next.js only. Add Express and Remix in V2 only after the interface is proven stable.
Entropy formula disagreement — teams reject the score as arbitrary
MED
Document the formula version, weights, and rationale explicitly (this spec). Allow teams to configure entropy weights in the spec under rules.entropyWeights. Treat the default weights as opinionated defaults, not laws.
Snapshot storage grows unbounded in large repos
LOW
Default retention of 50 snapshots with auto-prune. Snapshots are typically <10KB each. Configurable via spider.snapshotRetention.
Invariant compilation is underspecified in MVP
LOW
In MVP, invariants in the spec are treated as documentation only — not compiled to checks. Compiling them is a V3 feature. Document this clearly to set expectations.
§ 13
Open Decisions
These questions are unresolved and require a decision before implementation begins. Each is listed with the tradeoffs.

1. Spec file format: JSON vs YAML vs TypeScript config
JSON is maximally readable and has great schema tooling. YAML allows comments, which may be valuable in the spec. TypeScript config (like ESLint's eslint.config.ts) allows programmatic rules but increases onboarding friction. Recommendation: JSON for MVP. Revisit after real adoption reveals whether comments are missed.

2. Structure source resolution: filesystem crawl vs manifest injection
Filesystem crawl (scanning for route files) is zero-config but brittle — it must understand each framework's file conventions. Manifest injection (the build tool writes a spider.routes.json) is explicit and reliable but requires a plugin per framework. Recommendation: filesystem crawl for MVP with Next.js only. Add manifest injection in V2.

3. Snapshot storage: local files vs remote/shared store
Local files in .spider/snapshots/ committed to git is simple, auditable, and offline. A remote store enables team-wide history and cross-repo analysis but adds infrastructure and auth. Recommendation: local files only. Remote store is a commercial product feature, not an open-source feature.

4. Rule severity overrides in spec
Should teams be able to downgrade an ERROR to WARN, or suppress specific rules? This is how ESLint works and teams expect it. But it creates escape hatches that undermine enforcement. Recommendation: allow WARN downgrade and suppression for all rules except SPI-003, SPI-004, SPI-005 (the hard structural invariants). These three cannot be suppressed.

First commit goal
Day one of implementation: spider check runs successfully against a Next.js App Router project and outputs at least one real violation. Everything else is secondary to this. Ship a working check before touching graph, entropy, or evolution.
